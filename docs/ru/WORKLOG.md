### 2026-07-02 — Fix: веб-загрузка файла падала с 500 (TypeError на типизированном `UploadForm::$file`)
- **Симптом:** загрузка файла в админке (`POST /import/upload`) возвращала 500. В логе было `TypeError: Cannot assign string to property app\models\UploadForm::$file of type ?yii\web\UploadedFile`.
- **Причина:** `Html::activeFileInput()` рендерит **скрытое пустое поле `file`**, поэтому реальная отправка из браузера несёт `file=""` в теле POST (сам файл находится в `$_FILES`). Затем `Model::load()` пытался массово присвоить эту пустую **строку** **типизированному** свойству `?UploadedFile` → TypeError. (Консольный путь импорта это не затрагивало — он никогда не проходит через форму, поэтому и продолжал работать.)
- **Фикс:** `UploadForm::load()` теперь убирает ключ `file` перед массовым присваиванием; файл привязывается из `$_FILES` через `UploadedFile::getInstance()` в контроллере, как и раньше. Свойство осталось типизированным.
- **Регрессионный тест:** `UploadFormTest` — загрузка `['UploadForm' => ['source' => …, 'file' => '']]` теперь успешна и оставляет `file` равным null (раньше выбросила бы исключение). Полный набор **94 pass**, PHPStan 0, PHPCS без замечаний.
- **Проверено вживую** по публичному URL реальными multipart-отправками (скрытое пустое поле + часть с файлом): валидная загрузка CSV/JSON → 302 на keywords пакета; отправка **без файла** → 302 обратно с флеш-сообщением валидации (не 500). Демо-данные восстановлены 378 → 126.

- **Безопасно на границе.** Текст ключевого слова санируется до попадания в файл (валидный UTF-8, управляющие символы отброшены, пробелы схлопнуты, приведено к нижнему регистру, порядок слов сохранён — в отличие от отсортированного по токенам `normalized_term`), а дублирующиеся термины внутри группы схлопываются. Записываются только объявления с флагом `is_valid`; ad group без валидного объявления всё равно экспортирует свои ключевые слова и выводится как предупреждение в превью. Вывод — RFC-4180 (через запятую, в кавычках `"` с удвоением внутренних кавычек, CRLF), UTF-8 без BOM — кодировка, которую импортирует Editor.
- **Проверено вручную:** `yii export/file` пишет CSV на **127 строк** (заголовок + 107 строк keyword + 19 строк ad) по 6 campaigns / 19 ad groups, тип соответствия Phrase, локализованные Final URLs, UTF-8 сохранён (например, `Site.pro — DE`). Аутентифицированная веб-проверка: `/export` рендерится (200) с правильными счётчиками (6 / 19 / 107 / 19), а `/export/download` возвращает `text/csv` вложением (`google-ads-editor-<date>.csv`). **10 новых unit-тестов** (схема заголовка, дефолт Phrase + санация keyword вкл. управляющие символы, распределение по потолку RSA, экранирование RFC-4180 запятых/кавычек, CRLF) — полный набор unit-тестов **83 pass**. PHPCS без замечаний на новых файлах.

### 2026-07-01 — Этап 6: генерация объявлений (stored/template + валидация RSA)
- **Одно responsive search ad на ad group** (`services/adgen/`, миграция `generated_ad` + `AdGroup.generatedAd`). Для каждой группы сервис подбирает текст, применяет авторитетный целевой URL группы, валидирует и сохраняет. Админ-превью на `/ads`, консольный паритет `yii adgen/run`.
- **Два источника, приоритет корректности шаблона.** `TemplateAdGenerator` — детерминированный движок с языковыми строительными блоками (en/de/es/fr/it/pt), который вплетает собственный тематический токен ad group в пару заголовков и заполняет остальное локализованными ценностными предложениями конструктора сайтов — всегда на языке группы, всегда валидно, без внешних вызовов. `StoredAdSource` загружает более качественный текст, написанный офлайн (закоммиченный JSON с ключом `language:theme_key`), и **предпочитается, когда присутствует и валиден** (решение 3); всё отсутствующее или не прошедшее валидацию откатывается к шаблону. Так развёрнутый хост не выполняет генерацию и не держит AI-credentials.
- **Весь текст — недоверенный ввод.** `RsaValidator` применяет лимиты RSA от Google перед сохранением чего-либо: 3–15 заголовков ≤30 символов, 2–4 описания ≤90 символов, все различные, валидный UTF-8 без управляющих символов. **Целевой URL берётся из ad group, никогда из сгенерированного текста**, поэтому плохая строка не может перенаправить кампанию. Опциональные display paths проверяются по длине и содержимому.
- **Генерация — хвост пайплайна.** `generated_ad` полностью производный (перестраивается каждый прогон, идемпотентно) и FK-CASCADE на `ad_group`, поэтому повторная подготовка перестраивает группы и удаляет их объявления — **повторная подготовка инвалидирует этап 6 по замыслу**, ровно как повторная очистка инвалидирует этап 5 (решение 20). Это разрешило связку этапов 5/6, которую план оставлял открытой: `GroupingService` отказался от сохранения групп `ad_ready` в пользу простой полной перестройки (решение **27 заменяет 26**), что оставляет математику воронки этапа 5 нетронутой и избегает коллизий `theme_key`.
- **Проверено вручную:** миграция применяется; 19 ad groups → **19 ads** (6 из сохранённого EN-текста, 13 шаблонных), **0 invalid**, покрывая все **107** подготовленных keywords; консоль и веб совпадают. Повторная подготовка сохраняет 107/19 **и** каскадно обнуляет объявления до 0; повторная генерация восстанавливает 19 (три цикла стабильны). Страница `/ads` рендерится аутентифицированно (200, без ошибок) с посимвольными счётчиками заголовков/описаний и разбивкой stored/template. **22 новых unit-теста** (RsaValidator, TemplateAdGenerator, StoredAdSource) — полный набор unit-тестов **72 pass** на момент сборки.
- **Состязательный код-ревью** (5 линз — межэтапность/идемпотентность, RSA/недоверенный ввод/XSS, шаблонный генератор, числа в UI, документация; каждая находка проверена опровержением, большинство воспроизведено эмпирически в контейнере). Три линзы вернулись чистыми; две подтверждены и исправлены: (1) шаг ограничения в `TemplateAdGenerator` не отсекал управляющие символы, поэтому тема, несущая случайный управляющий байт (из неочищенного исходного ключевого слова), могла породить заголовок, не прошедший RSA-валидатор — теперь он отклоняет неочищенные элементы, восстанавливая инвариант "вывод всегда проходит RsaValidator" (добавлен регрессионный тест → **23 теста этапа 6, полный набор 73 pass**); объявление уже падало безопасно (сохранялось `is_valid=false`, без краша/XSS). (2) Подсказка `clean/run` говорила, что она только "сбрасывает этап 5", и предписывала перезапускать только подготовку, но CASCADE этапа 6 также очищает объявления — исправил сообщения консоли + веба для очистки/подготовки и документацию API/CLAUDE, чтобы связать re-clean → re-prepare → re-generate.

### 2026-07-01 — Этап 3: импорт + модель keyword + админ-GridView

- **Админ-область** (за входом): форма загрузки + сводка по источникам + история импорта, и полный GridView ключевых слов с фильтрами (source, language, stage, min volume), сортировкой и пагинацией.
- **Auth → env:** имя/пароль админа теперь берутся из `.env` (`ADMIN_USERNAME` / `ADMIN_PASSWORD`); никаких учётных данных, зашитых в PHP. `docker-compose.yml` читает всю конфигурацию из `.env` (см. `.env.example`); bcrypt-хэш вычисляется только на пути входа.
- **Консольный паритет:** `yii import/samples` и `yii import/file <source> <path>` выполняют тот же сервис (доказывает, что он развязан с веб-слоем).
- **Проверено вручную:** миграции применяются при старте контейнера; все четыре примерных файла импортируются (52 + 78 + 136 + 112 = 378 keywords; одна строка Search Console с пустым запросом пропущена по замыслу); гость → админ-страницы перенаправляют на вход; вход с учётными данными из `.env` работает; JSON-загрузка через веб-форму импортируется; GridView рендерится, фильтрует и сортирует (скриншоты).
- **Постсборочное усиление** (из состязательного код-ревью пути импорта, каждый фикс проверен против специально созданных входов): устойчивый разбор чисел — десятичные, научная нотация, разделители тысяч обрабатываются; неоднозначные значения (`12,50`, `1.2K`) отклоняются как null вместо тихого неверного масштабирования; колонки счётчиков расширены до `bigint`, поэтому одно значение за пределами диапазона больше не прерывает файл; ячейки CSV не в UTF-8 (Windows-1252) конвертируются, чтобы строка импортировалась и её аудит-копия `raw_data` сохранялась; дублирующиеся заголовки CSV отклоняются с понятным сообщением; проверка обязательных колонок использует объединение ключей по всем строкам (корректно для гетерогенного JSON); фильтры термина и конкуренции сетки ключевых слов сделаны функциональными; и ключ аутентификации remember-me выводится на каждый деплой вместо поставки общей закоммиченной константы.
